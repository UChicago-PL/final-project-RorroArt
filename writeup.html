A Compiler for a VLIW+SIMD Accelerator

The purpose of computing is insight, not numbers.

— Richard Hamming

Introduction

This document describes the design of a small optimizing compiler, written in Haskell, that targets a VLIW+SIMD accelerator. The compiler takes a fixed kernel — a program that hashes and traverses a forest data structure — and transforms it through a series of pure functional passes into a flat list of bundles, each representing one cycle of parallel work on the target machine. Along the way the compiler performs constant folding, automatic vectorization, peephole optimization, instruction lowering, and VLIW instruction packing.

The entire compiler is 1,400 lines of Haskell across 11 modules. Every pass is a pure function from one representation to the next. There are no mutable data structures, no IO in the compilation pipeline, and no monadic plumbing beyond a small State monad in two places (the IR builder and the lowering pass). The result is a compiler whose passes compose like ordinary functions:

let fn   = scalarKernel rounds batchSize
    fn'  = combineMulAdd . vectorizeBatch 8 . canonicalize $ fn
    ops  = lowerToMachineOps 8 fn'
    bundles = bundleProgram ops

Each of these names is a pure function, and the whole pipeline is an expression whose value is the final program. This is a very Haskell way to build a compiler, and it’s worth understanding both why it works and what each piece does.

The Target Machine

Before discussing the compiler, we should understand the machine it targets. The accelerator is a VLIW (very long instruction word) processor with SIMD (single instruction, multiple data) support. Each cycle, it can issue instructions to five independent execution engines in parallel:

EngineMax Slots/CycleOperations
ALU12Scalar arithmetic: +, -, *, //, ^, &, |, <<, >>, %, <, ==
VALU68-wide SIMD versions of the above, plus vbroadcast and multiply_add
Load2load, load_offset, vload (8 consecutive words), const (immediate)
Store2store, vstore (8 consecutive words)
Flow1select, vselect, halt, pause, jumps

A single cycle can therefore issue up to 12+6+2+2+1 = 23 operations. The register file is a flat scratch-pad addressed by integer index (ScratchAddr). SIMD operations work on blocks of 8 consecutive scratch addresses. Memory is flat and word-addressable. Within a single bundle (one cycle), all reads see the state from the beginning of the cycle, and all writes commit atomically at the end — there is no intra-bundle forwarding.

The compiler’s job is twofold: (a) vectorize scalar code to use 8-wide SIMD where legal, and (b) pack independent operations into each bundle to exploit the parallelism of the VLIW execution model.

The ISA is defined as a shared Haskell library (src/ISA.hs) that both the compiler and the simulator import. This means the compiler constructs typed ISA values — actual AluSlot, ValuSlot, LoadSlot values — and the simulator pattern-matches on the same constructors to execute them. If someone adds a new AluOp variant, both the compiler and simulator get non-exhaustive pattern warnings until they handle it. The types guarantee that the compiler can only emit instructions the simulator knows how to execute.

The Lifecycle of a Program

A program passes through seven stages, each a pure function that transforms the previous stage’s output:

  scalarKernel        -- build the IR            (BaselineKernel.hs)
      |
  canonicalize        -- constant fold           (Canonicalize.hs)
      |
  vectorizeBatch      -- auto-vectorize loops    (Vectorize.hs)
      |
  combineMulAdd       -- peephole: mul+add→madd  (Peephole.hs)
      |
  lowerToMachineOps   -- lower to machine ops    (FinalLowering.hs)
      |
  bundleProgram       -- VLIW bundle packing     (Bundle.hs)
      |
  renderProgramPayload -- emit Python literal    (PythonInterop.hs)

The first four stages operate on Function, the compiler’s intermediate representation. lowerToMachineOps converts to [MachineOp], a flat list of individual machine instructions with metadata. bundleProgram packs those into [ISA.Bundle ()], the final VLIW bundles. renderProgramPayload serializes the bundles as a Python dict literal that the test harness evaluates.

Let us now examine each stage in detail.

The Intermediate Representation (FinalIR.hs)

The IR is the heart of the compiler. It is a small structured-SSA language with three key design properties:

1. SSA values. Every Let introduces a fresh Id that is defined exactly once and used by later statements through Expr = Var Id | Const Word32. This is the static single assignment discipline: each name is assigned exactly once, and that definition dominates all uses. This makes dataflow analysis trivial — to find where a value comes from, you look up its (unique) definition.

2. Structured control flow. The only looping construct is For, which carries its bounds and body inline. There are no arbitrary gotos, no basic blocks, no phi nodes. The IR is a tree, not a graph. This means loop transformations are tree rewrites rather than graph surgeries.

3. Named memory regions. Loads and stores address a Buffer (a symbolic memory region) at an Expr offset, instead of raw pointer arithmetic. This is what makes the IR analyzable.

Types

data Ty = I32 | Vec !Int !Ty

I32 is a scalar 32-bit integer. Vec w I32 is a vector of w scalars, corresponding to w consecutive scratch-pad slots at the machine level. The kernel starts entirely scalar; the vectorizer introduces Vec types.

There is no Ptr, no Mask, no Mem token type. Pointers are represented as I32 values. Memory ordering follows statement order. Boolean conditions are I32 values (0 = false, nonzero = true). This simplicity is a deliberate choice, and we will examine the alternative — a richer type system with pointer types and memory tokens — later in this document.

Buffers: Symbolic Memory

data Buffer = Header | Forest | InpIdx | InpVal

These four constructors correspond 1:1 to the memory regions the test harness allocates:

BufferHarness RegionContents
Headerheader wordsrounds, n_nodes, batch_size, forest_height, 3 base pointers
Forestforest_valuesTree node array
InpIdxinp_indicesCurrent index for each batch element
InpValinp_valuesCurrent value for each batch element

Using symbolic buffer names instead of raw pointers is the single most important design decision in the IR. It is what makes the vectorizer possible without a general-purpose alias analysis. The vectorizer can ask, “Does this load use the induction variable as an index into InpIdx?” by pattern-matching on the buffer name. It never needs to solve the undecidable problem of whether two arbitrary pointer expressions might alias.

Load and Store Kinds

data LoadKind  = LoadScalar | LoadContigVec !Int | LoadGatherVec !Int
data StoreKind = StoreScalar | StoreContigVec !Int

The Kind tags record how wide a memory access is. The kernel emits only LoadScalar/StoreScalar. The vectorizer rewrites eligible accesses to LoadContigVec w (contiguous vector load of w words), LoadGatherVec w (indexed gather of w words at data-dependent offsets), or StoreContigVec w. The lowering then selects the correct machine instruction based on this tag: load vs. vload vs. load_offset, store vs. vstore.

This is an instance of a useful compiler design pattern: the analysis pass (Vectorize) communicates its decisions to the code-generation pass (Lowering) declaratively, by rewriting the IR, rather than through side channels. Each pass is a pure function from Function → Function, and the information flows through the data.

Right-Hand Sides

Every value-producing computation is one Rhs variant:

ConstructorMeaningIntroduced by
RConst w32-bit literalKernel
RBin op a bBinary ALU operationKernel
RMulAdd a b cFused multiply-add: a*b + cPeephole pass
RSelect c a bConditional select: c ≠ 0 ? a : bKernel
RLoad kind buf ixLoad from buf[ix]Kernel; kind rewritten by Vectorize
RVBroadcast w eBroadcast scalar to vector(Available for future use)

RBin takes AluOp directly from the shared ISA.hs library — the same enum the simulator uses. This means the compiler and simulator agree on the set of operations by construction.

RSelect replaces traditional if-conversion. Instead of if c then a else b as a control-flow construct, the kernel uses RSelect directly, which lowers to the simulator’s select (scalar) or vselect (vector) flow slot. This avoids the need for a separate if-conversion pass entirely.

RMulAdd does not appear in the initial kernel. It is introduced by the peephole pass when it discovers a Mul followed by an Add. The lowering maps it to a single multiply_add VALU slot for vectors, or two ALU slots (mul then add) for scalars.

Statements

data Stmt
  = Let !Id !Ty !Rhs
  | Store !StoreKind !Buffer !Expr !Expr
  | For !Id !Int !Int !Int ![Stmt]   -- iv, start, end, step, body

Let introduces a new SSA value. Store is a side effect. For is a counted loop with compile-time integer bounds and a step. The body is an inline list of statements, not a reference — the IR is a tree. Statement order determines memory ordering. There are no explicit memory tokens or effect edges. This is sound because all loops are fully unrolled during lowering, and within a single unrolled iteration the statement sequence is the execution sequence.

The Builder Monad

newtype Build a = Build { unBuild :: State BuildSt a }

emitLet   :: Ty -> Rhs -> Build Expr      -- emit Let, return the Var
emitStore :: StoreKind -> Buffer -> Expr -> Expr -> Build ()
emitFor   :: Int -> Int -> Int -> (Expr -> Build ()) -> Build ()

Build is a thin wrapper around State that provides a convenient monadic API for constructing IR. emitLet allocates a fresh Id, appends a Let statement, and returns the new Var as an Expr.

The interesting piece is emitFor: it takes a callback that receives the induction variable as an Expr and builds the loop body. Internally, capture temporarily redirects the statement buffer so that body statements don’t escape into the outer scope — a common technique for structured-SSA builders. The body is captured, the outer buffer is restored, and the captured body is wrapped in a For node. This is the kind of thing that Haskell’s first-class functions and closures make natural: the body-building callback closes over the induction variable and the enclosing environment, and the builder captures the side effects (emitted statements) without any of them leaking.

The Kernel (BaselineKernel.hs)

scalarKernel :: Int -> Int -> Function builds the baseline kernel using the Build monad. The two parameters are rounds (outer loop count) and batchSize (inner loop count). The result is a purely scalar Function with all types I32, all loads/stores scalar, and all loop steps equal to 1.

The kernel’s structure is:

1. Header loads (7 scalar loads from Header[0..6]): metadata including nNodes (the tree size, used for wrap-around) and base pointers for each memory region. The pointer fields are loaded but unused in the IR — the lowering resolves buffer names to header slots directly.

2. Outer loop over rounds.

3. Inner loop over the batch: for each element i:

  • Load idx from InpIdx[i] and val from InpVal[i].
  • Load the tree node from Forest[idx].
  • XOR val with the node and hash the result through 6 stages.
  • Compute the next index: idx*2 + (1 if hash%2==0, else 2).
  • Wrap: if nextIdx ≥ nNodes then nextIdx = 0.
  • Store the new index and hashed value back.

4. Hash function (emitHash): The 6-stage integer hash uses a table of (op1, const1, op2, op3, const3) tuples. Each stage computes a = op2(op1(a, const1), op3(a, const3)), matching the Python harness exactly. This is a nice example of Haskell’s expressiveness: the entire hash pipeline is a foldl over the stage table, with each step emitting IR via the builder monad.

emitHash :: Expr -> Build Expr
emitHash a0 = foldl go (pure a0) hashStages
  where
    go ma (op1, c1, op2, op3, c3) = do
      a  <- ma
      t1 <- emitLet I32 (RBin op1 a (Const c1))
      t2 <- emitLet I32 (RBin op3 a (Const c3))
      emitLet I32 (RBin op2 t1 t2)

Note the use of foldl over a list of stage descriptors, with the accumulator being a monadic computation (Build Expr). Each step sequences the monadic action from the previous stage with two new let-bindings and their combination. The <- extracts the value from the accumulated monadic computation, and emitLet adds to it. This is a pattern that arises frequently when building IR in a functional style: the monad provides sequencing and naming, while higher-order functions provide structure.

Pass 1: Canonicalize (Canonicalize.hs)

The first pass performs constant propagation and constant folding in a single forward traversal.

The mechanism is straightforward: goBlock walks statements in order, maintaining an environment Map Id Word32 of SSA values known to be constant. For each Let:

  1. simpRhs substitutes known-constant Vars with Const in the RHS operands.
  2. rhsConst checks whether the simplified RHS is fully constant (e.g., RBin Add (Const 3) (Const 5) folds to RConst 8).
  3. If so, the Id is added to the constant environment for use by later statements.

evalAluOp implements all 13 AluOp variants in 32-bit modular arithmetic, matching the simulator’s semantics. This function is defined once and is correct by construction: it handles every constructor of AluOp, and a non-exhaustive pattern warning would fire if a new operation were added to ISA.hs without updating the evaluator.

For For loops, the induction variable is deleted from the constant map when entering the body, since it takes different values on each iteration. Constants defined outside the loop remain visible inside.

For RSelect, if both branches are the same expression, the condition is replaced with Const 1 — a signal to later passes that the select is trivially dead.

Why does this pass matter for what follows? Canonicalization ensures the vectorizer sees clean operands rather than chains of trivially-constant lets. It also means the lowering’s constant-materialization cache has fewer unique constants to emit, reducing code size.

Pass 2: Vectorize (Vectorize.hs)

This is the core optimization. It rewrites eligible inner loops from scalar (step 1) to SIMD (step w, typically 8), turning scalar loads/stores into vector loads/stores and promoting arithmetic to vector width.

Legality

The vectorizer must first determine whether it is legal to vectorize a loop. The answer is yes only if the induction variable iv is used exclusively as a direct index into InpIdx or InpVal. Concretely, iv may appear in:

  • RLoad LoadScalar InpIdx (Var iv) — loading from inp_indices at the batch index
  • RLoad LoadScalar InpVal (Var iv) — loading from inp_values at the batch index
  • Store StoreScalar InpIdx (Var iv) _ — storing to inp_indices
  • Store StoreScalar InpVal (Var iv) _ — storing to inp_values

And it must not appear in arithmetic operands, non-InpIdx/InpVal loads, or nested loops.

This is a conservative map vectorizer: it vectorizes loops where each iteration is independent and the induction variable maps to a contiguous memory range. The check is purely syntactic — it pattern-matches on Buffer and Expr instead of doing a general dependence analysis. Named Buffer regions are what make this possible. Without them, the vectorizer would need to prove that two pointer expressions don’t alias, which is undecidable in general.

Strip-Mining and Tail Loops

When the trip count is not divisible by the SIMD width w, the vectorizer performs strip-mining:

vecEnd = start + (tripCount / w) * w

For iv start  vecEnd w  [vectorized body]    -- main loop
For iv vecEnd end    1  [scalar body]        -- tail loop

If vecEnd ≤ start (trip count < w), no vectorization occurs. If vecEnd == end (evenly divisible), no tail loop is emitted. For the default batchSize=256 with w=8, the trip count is exactly divisible, so no tail is needed. For batchSize=10, the main loop processes elements 0–7 and the tail processes 8–9.

Rewriting

Within the vectorized loop body, the vectorizer rewrites each statement:

Loads. RLoad LoadScalar InpIdx (Var iv) becomes RLoad (LoadContigVec w) InpIdx (Var iv). The load was reading one element at the batch index; now it reads w contiguous elements. The forest load RLoad LoadScalar Forest idx becomes RLoad (LoadGatherVec w) Forest idx when its index has been promoted to a vector, because the forest indices are data-dependent and therefore not contiguous.

Stores. Store StoreScalar InpIdx (Var iv) v becomes Store (StoreContigVec w) InpIdx (Var iv) v.

Arithmetic. RBin, RSelect, RMulAdd are structurally unchanged. The type is promoted by inferTy: if any operand is a Vec, the result is Vec w I32. Otherwise I32.

This propagation is what causes vector infection: once InpIdx/InpVal loads become vectors, every downstream computation — XOR, all six hash stages, mod, select, multiply — gets promoted too. The forest load becomes a gather because its index (which came from the now-vectorized InpIdx load) is a vector.

What the Vectorizer Does Not Do

The vectorizer does not handle reductions (loop-carried dependencies). It does not vectorize loops with nested loops. It does not handle strided access patterns. It does not reorder statements. These are deliberate simplifications. The baseline kernel’s inner loop is a pure map (no cross-iteration state), so the simple check is sufficient.

Pass 3: Peephole (Peephole.hs)

The peephole pass scans the statement list for adjacent pairs:

Let t (Vec w I32) (RBin Mul a b)
Let y (Vec w I32) (RBin Add (Var t) c)    -- or (RBin Add c (Var t))

where t is not used anywhere after y. It replaces both with:

Let y (Vec w I32) (RMulAdd a b c)

This is vector-only because the simulator has a multiply_add VALU instruction but no scalar fused-multiply-add. The vector version saves a VALU slot and a scratch register.

The dead-use check (usesId) traverses all subsequent statements, including nested loop bodies, to verify t doesn’t appear. This is conservative but sufficient for the straight-line hash pipeline.

The Machine Model (Machine.hs)

After the IR passes, the program is lowered to a sequence of MachineOp values. These bridge the gap between the IR (which knows about buffers and types) and the ISA (which knows about scratch addresses and instruction slots).

data MachineSlot
  = MSlotAlu   !AluSlot
  | MSlotValu  !ValuSlot
  | MSlotLoad  !LoadSlot
  | MSlotStore !StoreSlot
  | MSlotFlow  !FlowSlot

data MachineOp = MachineOp
  { moSlot   :: !MachineSlot
  , moReads  :: ![Int]       -- scratch addresses read
  , moWrites :: ![Int]       -- scratch addresses written
  , moMem    :: ![MemAccess] -- memory regions accessed
  }

Each MachineSlot wraps one of the typed slot ADTs from ISA.hs. For example, MSlotAlu (Alu Add (ScratchAddr 10) (ScratchAddr 5) (ScratchAddr 6)) represents a scalar add writing scratch 10. The slot carries full type information — not strings — so the bundler extracts typed slot lists directly into the final ISA.Bundle.

The metadata fields (moReads, moWrites, moMem) are what the bundler uses to determine which operations can safely share a cycle. For a scalar op, moWrites is a single address; for a vector op, it’s w consecutive addresses. MemAccess records describe the memory region, direction (load/store), and optional address range for each memory operation. An Unknown range conservatively prevents packing with any other access to the same region.

Lowering (FinalLowering.hs)

The lowering pass is the largest module (478 lines) because it handles scratch-pad allocation, address computation, constant materialization, loop unrolling, and the scalar/vector dispatch for every Rhs variant.

Scratch-Pad Layout

The lowering allocates scratch-pad addresses in a single up-front pass:

 Address      Purpose
 ──────────   ────────────────────────────────────
 0..7         Header block (loaded by a single vload in the prelude)
 8            Address temp for InpIdx / Forest / Header
 9            Address temp for InpVal
 10..17       Address temp vector (for gather base+index computation)
 18..         SSA value storage (1 slot per I32, w slots per Vec)

Loop induction variables don’t get persistent slots — they are materialized as constants on each unrolled iteration. Header-load IDs are already mapped to header block slots 0–6.

Header Prelude

The lowering emits a single vload that reads 8 words from memory address 0 into scratch slots [0..7]. This loads the entire header in one cycle. The IR’s seven RLoad LoadScalar Header statements are then no-ops during lowering — the value for each is already in its scratch slot.

Constants and Broadcasts

Three caches avoid redundant materialization:

CacheKeyInstruction Emitted
lsConstSWord32const dst val (Load engine)
lsConstVWord32vbroadcast dst src (VALU engine)
lsBcastScalar scratchvbroadcast dst src (VALU engine)

getConstS c materializes a scalar constant once and caches the scratch address. getConstV c calls getConstS then broadcasts. getBroadcast s broadcasts an existing scalar scratch to a vector. All three check their cache first and emit nothing on a hit. When a vector operation has a scalar operand (e.g., a Const in the IR), the lowering calls exprVector which automatically broadcasts it.

Scalar vs. Vector Dispatch

lowerLet switches on the type of the let-binding:

IR TypeRBin opRSelectRLoad
I32Alu op (ALU)Select (Flow)Load / load (Load)
Vec _ _VAlu op (VALU)VSelect (Flow)VLoad / load_offset (Load)

For vectors, moReads/moWrites lists span w consecutive addresses. For scalars, they are single-element lists.

Address Computation and Caching

Memory accesses go through computeAddr, which computes base_ptr + offset. The function basePtr maps each Buffer to its base scratch address: Header → const 0, Forest → hdrBase+4, InpIdx → hdrBase+5, InpVal → hdrBase+6. These are the pointer words from the header.

An optimization: if the same buffer was just accessed at the same static offset, the address ALU add is skipped and the cached address temp is reused. This saves an ALU slot when the same address is used for a load and its subsequent store (common: InpIdx[i] is loaded and stored in every iteration).

Gather Loads

lowerLoadVecGather handles the forest load after vectorization. The index is a vector of w data-dependent offsets:

  1. Broadcast the forest base pointer to a vector.
  2. VALU-add the index vector and base vector, producing a vector of absolute addresses.
  3. Issue w individual load_offset instructions, one per lane.

Each load_offset reads from a different element of the address vector. The memory metadata is Unknown because gather addresses are data-dependent.

Loop Unrolling

The lowering fully unrolls every loop by iterating [start, start+step .. end-1] and emitting the body for each iteration value. This is valid because loop bounds are compile-time constants. Before each iteration, if the IV is used in the body, a const load materializes the current iteration value as a scalar scratch.

After vectorization, the inner loop has step w=8. For batchSize=256, this means 32 unrolled iterations (down from 256 scalar iterations). Each iteration emits vector ops that process 8 elements, so the total work is the same but uses roughly 8x fewer instructions.

Bundling (Bundle.hs)

The bundler takes the flat [MachineOp] list and packs it into VLIW bundles, where each bundle represents one cycle on the simulator.

Algorithm

The bundler is a greedy in-order packetizer: it scans ops front-to-back and tries to add each op to the current bundle. If the op doesn’t fit, the current bundle is flushed and a new one is started. Ops are never reordered.

This is not optimal (a scheduler that considers latencies could do better), but it is simple, correct, and still achieves significant packing because the lowering emits operations in a natural order where adjacent ops often target different engines.

Legality Checks

canPack checks four conditions. All must hold for the op to enter the current bundle:

1. Slot capacity. Each engine has a maximum: ALU ≤ 12, VALU ≤ 6, LOAD ≤ 2, STORE ≤ 2, FLOW ≤ 1.

2. No RAW hazard. The op’s reads must not overlap with addresses written by earlier ops in this bundle. Even though the simulator executes all ops in a bundle atomically, the compiler emits ops in program order and expects sequential semantics.

3. No WAW hazard. No two writes to the same scratch address in one cycle.

4. No memory hazard. If the op accesses a memory region that already has a store in this bundle, the address ranges must not overlap.

Packing in Practice

For a batch size of 16 with 1 round, the test suite shows: scalar pipeline produces 386 bundles (nearly one op per cycle), while the vectorized pipeline produces 77 bundles. The 5x reduction comes from two sources: vectorization reduces instruction count by ~8x, and the bundler packs multiple independent ops into each cycle (an ALU address computation and a LOAD can share a cycle, two LOADs can share a cycle, a VALU and a LOAD const can share a cycle, and so on).

Rendering (PythonInterop.hs)

renderProgramPayload produces a Python dict literal:

{"rounds":3,"batch_size":256,"program":[{...},{...},...]}

Each bundle is a dict with engine keys ("alu", "valu", "load", "store", "flow") mapping to lists of tuples. The rendering functions pattern-match on the typed ISA slot constructors and produce the correct string. Because the slots are typed, the renderer is total over all ISA constructors and the output format is guaranteed to match what the simulator’s parser expects.

Worked Example: One Batch Element Through the Pipeline

To make the pipeline concrete, let us trace the transformation of one batch element through every stage.

Stage 1: Scalar IR

The inner loop body for element i (all types I32):

Let %12 I32 (RLoad LoadScalar InpIdx (Var %10))      -- idx = InpIdx[i]
Let %13 I32 (RLoad LoadScalar InpVal (Var %10))      -- val = InpVal[i]
Let %14 I32 (RLoad LoadScalar Forest (Var %12))      -- node = Forest[idx]
Let %15 I32 (RBin Xor (Var %13) (Var %14))           -- val ^= node
   ... 6 hash stages, each 3 Lets ...
Let %33 I32 (RBin Mod (Var %32) (Const 2))           -- mod2 = hash % 2
Let %34 I32 (RBin Eq_ (Var %33) (Const 0))           -- even?
Let %35 I32 (RSelect (Var %34) (Const 1) (Const 2))  -- add = even?1:2
Let %36 I32 (RBin Mul (Var %12) (Const 2))           -- idx*2
Let %37 I32 (RBin Add (Var %36) (Var %35))           -- nextIdx
Let %38 I32 (RBin Lt (Var %37) (Var %1))             -- < nNodes?
Let %39 I32 (RSelect (Var %38) (Var %37) (Const 0))  -- wrap to 0
Store StoreScalar InpIdx (Var %10) (Var %39)
Store StoreScalar InpVal (Var %10) (Var %32)

Stage 3: After Vectorize (step becomes 8)

The vectorizer checks: iv (%10) is used only as an index into InpIdx and InpVal. Legality passes. Types are promoted:

Let %12 (Vec 8 I32) (RLoad (LoadContigVec 8) InpIdx (Var %10))
Let %13 (Vec 8 I32) (RLoad (LoadContigVec 8) InpVal (Var %10))
Let %14 (Vec 8 I32) (RLoad (LoadGatherVec 8) Forest (Var %12))
Let %15 (Vec 8 I32) (RBin Xor (Var %13) (Var %14))
   ... all hash stages now Vec 8 I32 ...
Store (StoreContigVec 8) InpIdx (Var %10) (Var %39)
Store (StoreContigVec 8) InpVal (Var %10) (Var %32)

The InpIdx/InpVal loads became contiguous vector loads. The forest load became a gather (its index is now a vector). All arithmetic was promoted to Vec 8 I32. The (Const 2) operands remain I32 — the lowering will broadcast them.

Stage 5: After Lowering (one unrolled iteration at i=0)

MSlotAlu  (Alu Add addr=8 base=5 offset=const(0))  -- addr for InpIdx[0]
MSlotLoad (VLoad dst=18 addr=8)                     -- InpIdx[0:8] → scratch 18..25
MSlotAlu  (Alu Add addr=9 base=6 offset=const(0))  -- addr for InpVal[0]
MSlotLoad (VLoad dst=26 addr=9)                     -- InpVal[0:8] → scratch 26..33
MSlotValu (VAlu Add addrV=10 ixV=18 baseV=bcast(4)) -- gather addresses
MSlotLoad (LoadOffset dst=34 base=10 lane=0)        -- Forest[idx[0]]
...                                                  -- 8 lane loads total
MSlotValu (VAlu Xor dst=42 src=26 src=34)           -- val ^= node
... (6 hash stages, each ~3 VALU ops) ...
MSlotStore (VStore addr=8 src=...)                   -- InpIdx[0:8] = wrapped
MSlotStore (VStore addr=9 src=...)                   -- InpVal[0:8] = hashed

Stage 6: After Bundling

The bundler packs independent ops into cycles: the address ALU + VLoad share a cycle (different engines), the 8 gather LoadOffsets span 4 cycles (2 loads/cycle), VALU hash ops pack up to 6 per cycle, and the two VStores fit in one cycle.

An Earlier, More Ambitious Design

The current compiler was preceded by a significantly more complex design — approximately 3,500 lines across 14 modules, compared to the current 1,400 lines across 11. This earlier design attempted to build a general-purpose compiler infrastructure, and understanding what it tried to do and why those things are hard provides useful perspective on the tradeoffs inherent in compiler design.

The Old IR: Region-SSA with Parametric Debugging

The old FinalIR.hs (258 lines) defined a richly-typed IR with deep support for memory effects, affine indexing, SIMD execution modes, and parametric debugging. Its core differences from the current design were:

Type parameter k. Every type in the IR — Kernel k, Region k, Stmt k, Effect k — was parameterized over a phantom type k representing debug key references. This made the entire IR a Functor over debug key representations, enabling the compiler to transform debug annotations independently of program logic. The cost was that every function touching the IR had to be polymorphic in k, and Functor instances had to be derived for every type.

A richer type system. The old IR had five types instead of two: I32, Ptr, Vec Width Ty, Mask Width, and Mem MemSpace. Ptr was a first-class address type distinct from I32, requiring separate pointer-arithmetic validation in the lowering. Mask was a dedicated predicate vector type for SIMD conditionals. Mem MemSpace was a phantom token type for memory effect sequencing.

Affine index algebra. Rather than plain Expr offsets, the old IR had a full affine expression language:

data Ix = IxConst Int | IxVar Id | IxLane | IxAdd Ix Ix | IxMul Int Ix
data Index = IndexAff Ix | IndexVal Id
data Addr = Addr { addrBase :: Id, addrIndex :: Index }

IxLane represented the SIMD lane index. This algebra let the vectorizer reason about access patterns: IxConst + IxMul 1 (IxVar iv) is contiguous (vectorizable), IxMul 2 (IxVar iv) is strided (not vectorizable). The recursive lowering of this algebra to scratch-register operations took about 80 lines and handled constant folding at every node.

Memory tokens. Every load consumed a Mem token and produced both a value and a new Mem token. Every store consumed and produced a token. The Let node had multiple outputs [(Id, Ty)] to accommodate the value+token pair. The tokens had to be threaded linearly through control flow: in the presence of If nodes, the lowerer had to merge token chains from both branches.

Structured If with regions. The old IR had If { ifCond, ifThen :: Region k, ifElse :: Region k, ifOuts }. Each Region had regionParams, regionStmts, and regionYield — a structured form of phi-node that encoded which values flow into and out of the region. The For loop similarly had forInits (loop-carried values) and a region body with params and yields.

Carry parameters and reduction. The loop’s region params/yields encoded loop-carried state. The vectorizer had to analyze each carry to determine whether it was (a) a memory token (pass through), (b) an associative reduction like acc = acc + x[i], or (c) unsupported. RReduce AluOp Id was a dedicated IR node for horizontal SIMD reduction.

The Old Vectorizer: 998 Lines of Analysis

The old BatchVectorizeFinalIR.hs (998 lines) was a sophisticated pass. It classified values as MUniform (same across lanes), MLaneBase (contiguous with the induction variable), or MVec (fully vector). It performed backward stride analysis through the affine index algebra. It had a prepass that converted If nodes into RSelect operations (if-conversion), and another prepass that canonicalized address expressions to recover structured information from flattened pointer arithmetic.

Compare this with the current vectorizer: 156 lines, a single legality check (ivOnlyUsedAsInpIndex), and a straightforward rewrite of load/store kinds.

The Old Lowering: 1,224 Lines

The old lowering had to handle If nodes (with conditional jumps and branch merging), Region with carry parameters, Ptr type validation, recursive affine index lowering, memory token validation, multi-output Let nodes, a parametric debug key mapping function, and RReduce expansion into vector accumulation + horizontal reduction + scalar combine. The current lowering is 478 lines.

Why These Features Are Genuinely Hard

Each of the features that was removed represents a genuinely difficult compiler engineering problem:

Memory tokens enforce memory ordering through the type system. This is the approach used by MLIR’s MemRef dialect and LLVM’s memory SSA. The difficulty is composability: every load and store must consume and produce a token, and the tokens must be threaded linearly through control flow. In the presence of If nodes, the lowerer must merge token chains from both branches. A single dropped or duplicated token silently breaks memory ordering, causing loads to see stale values. The old compiler’s multi-output Let and carry parameter threading through Region yields was the mechanism for this, and it required careful validation at every step.

Affine index algebra provides the representation for stride analysis, but the analysis itself must propagate stride information backward through arbitrary expression DAGs. Getting this wrong means the vectorizer might emit a contiguous vload for what is actually a strided or random access pattern — a silent correctness bug that produces wrong results but no crash.

If-conversion (converting branches to predicated select operations) is sound only when both branches are pure. The boundary is subtle: a branch that looks pure but contains a load through an aliased pointer is not safe to speculate. The interaction with memory tokens makes this especially delicate.

Carry analysis and reduction vectorization require recognizing that a loop-carried dependence like acc = acc + x[i] can be vectorized by accumulating partial sums in a vector and reducing at the end. The analysis must verify associativity of the reduction operator, correctly identify which operand is the carry vs. the contribution, and generate the reduction epilogue. Non-associative operations (subtraction, division) must be correctly rejected.

Parametric debug policies introduced a higher-kinded abstraction where the compiler was generic over the representation of debug keys. When a scalar loop became a vector loop, one debug key per iteration had to become w keys per vector iteration. Error handling had to flow through Either String at every policy callback, and the policy interacted with every IR transformation.

The Tradeoff

The fundamental design shift was from a general-purpose region-SSA compiler to a domain-specific compiler for a known kernel shape. The old compiler was built to handle arbitrary structured programs with unknown access patterns, dynamic loop bounds, conditional branches, and loop-carried dependences. The new compiler exploits the fact that the target kernel has a fixed structure: nested For loops over known buffers with known access patterns.

The quantitative result is a 3.3x reduction in code size:

ComponentOldNewRatio
IR definition258 lines144 lines1.8x
Lowering1,224 lines478 lines2.6x
Vectorization998 lines156 lines6.4x
Kernel builder528 lines62 lines8.5x
Total~3,500 lines / 14 modules1,400 lines / 11 modules2.5x

This is a classic tradeoff in compiler design. The old design was more powerful and extensible. The new design is simpler, more transparent, and produces correct output for the actual workload. Neither is wrong — they are answers to different questions. The old compiler asked, “What would it take to handle any structured program on this machine?” The new one asks, “What is the clearest path from this program to efficient machine code?”

One way to view the relationship is that the features in the old compiler — memory tokens, affine indices, if-conversion, carry analysis — represent a menu of things a compiler might need. The insight behind the new design is that for this kernel, we only need a small subset of that menu, and by committing to that subset we gain enormous simplicity.

Haskell Techniques in the Compiler

The compiler is a good showcase for several Haskell idioms:

ADTs as the universal data structure. The IR, the machine model, the bundles — every intermediate form is an algebraic data type. Pattern matching provides exhaustive dispatch, and the compiler rejects ill-formed inputs at compile time rather than runtime.

Pure functions as passes. Every compiler pass is a pure function: Function → Function, Function → [MachineOp], [MachineOp] → [Bundle ()]. This makes it trivial to test passes in isolation, compose them in any order, and reason about what each one does.

The State monad for accumulation. The IR builder and the lowering pass use State to thread a counter (for fresh IDs / scratch addresses) and an accumulator (for emitted statements / instructions). This is a disciplined use of mutable state: the mutation is bounded within a runState call, and the result is a pure value.

Higher-order functions for structure. emitFor takes a callback that builds the loop body. emitHash folds over a table of stage descriptors. The vectorizer’s goBlock is a foldl over statements with an environment accumulator. These are not exotic techniques — they are the bread and butter of functional programming — but they lead to surprisingly concise compiler code.

Shared ADTs as a contract. The AluOp type, the slot ADTs, and the Bundle type from ISA.hs are shared between the compiler and the simulator. This is a compile-time guarantee that the compiler can only emit valid instructions. In a language without ADTs, this guarantee would require runtime validation or external schema checking.

Summary

The compiler transforms a scalar kernel into vectorized, bundled machine code through six pure functional passes:

  1. Kernel construction builds the scalar IR using a monadic builder.
  2. Canonicalization folds constants.
  3. Vectorization rewrites eligible loops to process 8 elements per iteration, with tail loops for non-divisible trip counts.
  4. Peephole optimization fuses multiply-add pairs.
  5. Lowering allocates scratch addresses, unrolls loops, and emits typed machine operations with hazard metadata.
  6. Bundling packs operations into VLIW bundles, checking slot limits, register hazards, and memory conflicts.

The whole pipeline composes into four lines of Haskell. Every intermediate form is an algebraic data type, every pass is a pure function, and the shared ISA library provides a compile-time contract between the compiler and the simulator. The result is a compiler that is small (1,400 lines), correct (all tests pass), and reasonably efficient (5x fewer cycles than the scalar baseline for the same computation).
